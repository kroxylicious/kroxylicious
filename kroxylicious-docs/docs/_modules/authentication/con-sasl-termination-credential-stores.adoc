= Credential stores

[role="_abstract"]
Credential stores provide pluggable credential sources for SASL Termination, allowing integration with various identity management systems.

== How credential stores work

When a client attempts SASL authentication:

1. The client sends their username as part of the SASL exchange
2. The SASL Termination filter extracts the username
3. The filter queries the configured credential store for that username
4. The credential store returns the stored credential (salt, iterations, server key, stored key)
5. The filter uses the credential to validate the client's authentication challenge

Credential stores are pluggable, allowing integration with different backend systems.

== KeystoreScramCredentialStore

The KeystoreScramCredentialStore implementation loads SCRAM credentials from Java keystores (PKCS12 or JKS format).

*Key features:*

* Loads credentials from standard Java keystore files
* Supports PKCS12 (recommended) and JKS formats
* In-memory operation for fast credential lookups
* File-based credential management using keystore-credential-tool CLI
* Support for both SCRAM-SHA-256 and SCRAM-SHA-512

*When to use:*

* Small to medium deployments (up to thousands of users)
* File-based credential management is acceptable
* No requirement for real-time credential updates
* Centralized credential distribution via configuration management

== Credential format

Credentials are stored as JSON-encoded SecretKey entries in the keystore:

[source,json]
----
{
  "username": "alice",
  "salt": "base64-encoded-bytes",
  "iterations": 10000,
  "serverKey": "base64-encoded-bytes",
  "storedKey": "base64-encoded-bytes",
  "hashAlgorithm": "SHA-256"
}
----

The keystore-credential-tool CLI manages this format automatically.
See xref:proc-managing-credentials-keystore-tool-{context}[] for credential management procedures.

== Future credential store implementations

The credential store API is extensible. Future implementations could integrate with:

* LDAP directories
* Relational databases
* External identity providers
* HashiCorp Vault or similar secret management systems
* Cloud-based identity services

Custom credential store implementations can be developed by implementing the `ScramCredentialStoreService` interface.
