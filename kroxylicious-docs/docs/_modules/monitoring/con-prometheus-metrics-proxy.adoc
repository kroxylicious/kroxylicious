= Proxy metrics overview

[role="_abstract"]

The proxy provides metrics for both connections and messages.
These metrics are categorized into downstream (client-side) and upstream (broker-side) groups
They allow users to assess the impact of the proxy and its filters on their Kafka system.

* Connection metrics count the connections made from the downstream (incoming connections from the clients) and the connection made by the proxy to upstream (outgoing connections to the Kafka brokers).
* Message metrics count the number of {kafka-protocol}#protocol_messages[Kafka protocol request and response messages] that flow through the proxy.

== Connection metrics

Connection metrics count the TCP connections made from the client to the proxy (`kroxylicious_client_to_proxy_request_total`) and from the proxy to the broker (`kroxylicious_proxy_to_server_connections_total`).
These metrics count connection _attempts_, so the connection count is incremented even if the connection attempt ultimately fails.

In addition to the count metrics, there are active connection gauge metrics that track the current number of open connections, and error metrics.

* If an error occurs whilst the proxy is accepting a connection from the client the `kroxylicious_client_to_proxy_errors_total` metric is incremented by one.
* If an error occurs whilst the proxy is attempting a connection to a broker the `kroxylicious_proxy_to_server_errors_total` metric is incremented by one.

Connection and connection error metrics include the following labels: `virtual_cluster` (the virtual cluster's name) and `node_id` (the broker's node ID).
When the client connects to the bootstrap endpoint of the virtual cluster, a node ID value of `bootstrap` is recorded.

The `kroxylicious_client_to_proxy_errors_total` metric also counts connection errors that occur before a virtual cluster has been identified.
For these specific errors, the `virtual_cluster` and `node_id` labels are set to an empty string ("").

NOTE: Error conditions signaled _within_ the Kafka protocol response (such as `RESOURCE_NOT_FOUND` or `UNKNOWN_TOPIC_ID`) are not classed as errors by these metrics.

=== Understanding connection metrics relationships

The proxy provides several related metrics for tracking connections:

* **Connection counters** (`kroxylicious_*_connections_total`) track the total number of connection attempts over time. These values only increase and provide a historical view of connection activity.
* **Active connection gauges** (`kroxylicious_*_active_connections`) show the current number of open connections at any given moment. These values increase when connections are established and decrease when connections are closed.
* **Error counters** (`kroxylicious_*_errors_total`) track connections that closed due to errors.
* **Disconnect counters** (`kroxylicious_client_to_proxy_disconnects_total`) track connections that closed without errors, categorized by cause.

When a connection closes, it increments either the error counter or one of the disconnect counter causes, but never both.
The active connection gauge decreases regardless of whether the closure was due to an error or a clean disconnect.

The following relationship holds:

`Active connections = Connections total - (Errors total + sum of all Disconnect causes)`

== Message metrics

Message metrics count and record the sizes of the Kafka protocol requests and responses that flow through the proxy.

Use these metrics to help understand:

* the number of messages flowing through the proxy.
* the overall volume of data through the proxy.
* the effect the filters are having on the messages.

* Downstream metrics
** `kroxylicious_client_to_proxy_request_total` counts requests as they arrive from the client.
** `kroxylicious_proxy_to_client_response_total` counts responses as they are returned to the client.
** `kroxylicious_client_to_proxy_request_size_bytes` is incremented by the size of each request as it arrives from the client.
** `kroxylicious_proxy_to_client_response_size_bytes` is incremented by the size of each response as it is returned to the client.

* Upstream metrics
** `kroxylicious_proxy_to_server_request_total` counts requests as they go to the broker.
** `kroxylicious_server_to_proxy_response_total` counts responses as they are returned by the broker.
** `kroxylicious_proxy_to_server_request_size_bytes` is incremented by the size of each request as it goes to the broker.
** `kroxylicious_server_to_proxy_response_size_bytes` is incremented by the size of each response as it is returned by the broker.

The size recorded is the encoded size of the protocol message. It includes the 4 byte {kafka-protocol}#protocol_common[message size].

Filters can alter the flow of messages through the proxy or the content of the message.
This is apparent through the metrics.

* If a filter sends a short-circuit, or closes a connection the downstream message counters will exceed the upstream counters.
* If a filter changes the size of the message, the downstream size metrics will be different to the upstream size metrics.

.Downstream and upstream message metrics in the proxy
image::{realimagesdir}/monitoring-message-counters.svg["Conceptual diagram showing the downstream and upstream message metrics within the proxy, illustrating how they respond to message transit through it."]

Message metrics include the following labels: `virtual_cluster` (the virtual cluster's name), `node_id` (the broker's node ID), `api_key` (the message type), `api_version`, and `decoded` (a flag indicating if the message was decoded by the proxy).

When the client connects to the bootstrap endpoint of the virtual cluster, metrics are recorded with a node ID value of `bootstrap`.

== Reconfiguration metrics

When the proxy applies a configuration change in place (a reconfiguration, without a restart), it emits metrics that let operators track how often reconfigurations happen, how long they take, and whether they succeed.
A reconfiguration may add, remove, or modify virtual clusters; removing or modifying a virtual cluster first drains its existing connections (see <<Virtual cluster lifecycle metrics>>).

Reconfiguration outcomes are reported through the `outcome` label on `kroxylicious_reconfigure_total`:

* `success` - every affected virtual cluster was reconfigured cleanly.
* `partial_failure` - the reconfiguration ran to completion but at least one virtual cluster failed (for example, a port could not be bound). Inspect `kroxylicious_reconfigure_clusters_affected_total` with `outcome="failure"` to see which operations failed.
* `catastrophic` - the reconfiguration could not be applied and the proxy retained its previous configuration.

Reconfigurations that are rejected before any change is attempted (a concurrent reconfiguration, or a change to a setting that cannot be altered at runtime) are not counted by `kroxylicious_reconfigure_total`.

.Reconfiguration metrics
|===
|Metric Name |Type |Labels|Description

|`kroxylicious_reconfigure_total`
|Counter
|`outcome`
|Incremented by one for every reconfiguration that is attempted, labelled by its overall `outcome`.

|`kroxylicious_reconfigure_duration_seconds`
|Timer
|
|Records the end-to-end duration of each reconfiguration.

|`kroxylicious_reconfigure_clusters_affected_total`
|Counter
|`operation`, `outcome`
|Incremented by one for each per-virtual-cluster operation during a reconfiguration. The `operation` label is `add`, `remove`, or `modify`; the `outcome` label is `success` or `failure`.

|`kroxylicious_drain_duration_seconds`
|Timer
|`virtual_cluster`
|Records how long it took to drain a virtual cluster's connections (when the cluster is removed, modified, or the proxy is shutting down). A consistently high value indicates long-lived connections; use it to tune the drain timeout.

|`kroxylicious_drain_connections_force_closed_total`
|Counter
|`virtual_cluster`
|Incremented by one for every connection that was force-closed because it had not drained before the drain timeout expired. A high rate suggests the drain timeout is too aggressive. This is the same event also recorded by `kroxylicious_client_to_proxy_disconnects_total` with `cause="drain_timeout"`.

|===

== Virtual cluster lifecycle metrics

Each virtual cluster moves through a well-defined lifecycle. These metrics expose its current state, how long it spends in each state, and the transitions between states, giving operators insight into start-up, reconfiguration, and shutdown behaviour.

The lifecycle states are:

* `initializing` - the virtual cluster is being set up and is not yet accepting traffic.
* `serving` - the virtual cluster is active and serving client traffic.
* `draining` - the virtual cluster is no longer accepting new connections; existing in-flight requests are being allowed to complete before the connections are closed. A cluster drains when it is removed or modified by a reconfiguration, or when the proxy is shutting down.
* `failed` - the virtual cluster could not be brought up (for example, its configuration was not viable) and its resources have been released.
* `stopped` - terminal state; the virtual cluster has been permanently removed.

`kroxylicious_virtual_cluster_state` is a state-set gauge: for each virtual cluster, exactly one `state` series reads `1` (the current state) and the others read `0`.
For example, to find clusters that failed to start, query `kroxylicious_virtual_cluster_state{state="failed"} == 1`.

.Virtual cluster lifecycle metrics
|===
|Metric Name |Type |Labels|Description

|`kroxylicious_virtual_cluster_state`
|Gauge
|`virtual_cluster`, `state`
|Reports the current lifecycle state of each virtual cluster. The series matching the current state reads `1`; all other states for that cluster read `0`.

|`kroxylicious_virtual_cluster_state_duration_seconds`
|Timer
|`virtual_cluster`, `state`
|Records how long a virtual cluster spent in a state, recorded when it leaves that state.

|`kroxylicious_virtual_cluster_transitions_total`
|Counter
|`virtual_cluster`, `from`, `to`
|Incremented by one for every lifecycle state transition, labelled with the `from` and `to` states.

|===
